# Dogfood — real-compaction-case

**Date:** 2026-07-02  
**Case:** `tests/dogfood/real-compaction-case.ts`  
**Invocation:** `REAL_COMPACTION_TURNS=6 REAL_COMPACTION_CONTEXT_WINDOW=4200 REAL_COMPACTION_THRESHOLD=0.35 pnpm tsx tests/dogfood/real-compaction-case.ts`  
**Model:** `deepseek-chat` main + `deepseek-chat` aux  

## Goal

Create a real-provider dogfood case that mimics a human long task around:

1. push `main`
2. update/install x_harness
3. use or create skills
4. run installed CLI dogfood
5. continue after compaction using the compacted session summary

This is intentionally integrated into the existing `x-harness-real-dogfood` SOP skill as `--case real-compaction --with-provider`, rather than creating one skill per case.

## Observed run

The 6-turn run triggered compaction from turn 3 onward:

The summary passed basic continuity checks:

- retained the skill/install/dogfood domain
- mentioned concrete artifacts and validation ideas
- preserved early-turn signal rather than only the final exchange
- did not become an instruction to execute tools

## Important UX finding

The compacted summary also showed a subtle but important failure mode:

> It preserved the right *kind* of details, but factualized illustrative examples as if they had happened.

Examples observed in the summary:

- `test_skill_v2.md`
- `x_harness validate`
- `wget`
- `events.jsonl`
- `successful_steps` / `failed_steps`

These were not concrete completed artifacts from the actual test conversation. They were plausible examples generated by the model while discussing design. If a later model resumed from this summary, it might waste time looking for files/commands that never existed, or treat speculative examples as audit truth.

## Test case change

The case now checks both:

1. **must-keep signals** — domain, paths, validation intent, early-turn signal
2. **must-not-invent signals** — concrete artifacts that were not actually introduced as completed state

This makes the test more useful than a simple "did compaction fire" check.

## Recommendations

### P0 — Add factuality discipline to compaction summaries

Update `FILTER_SAFE_PREAMBLE` so the summarizer distinguishes:

- facts that happened
- user requirements
- proposed examples
- unresolved suggestions

Suggested prompt addition:

> Do not turn illustrative examples or proposed filenames into completed facts. Mark speculative examples as "example/proposal" unless the transcript explicitly shows they were created, executed, or verified.

### P0 — Treat dogfood summary checks as quality gates for compaction changes

`real-compaction-case` should stay opt-in because it uses a real provider, but it should be run before major compaction prompt/config changes.

### P1 — Persist machine-readable dogfood reports

The script should eventually write a JSON report with:

- compaction events
- turn stats
- summary text
- pass/fail checks
- must-keep/must-not-invent terms

This allows comparing summary quality over time.

### P2 — Scenario-specific golden terms

Each long-task dogfood case should define:

- `mustKeep`: terms that must survive compaction
- `mustNotInvent`: terms that should fail the summary if presented as facts

This is better than one generic summary-quality regex for all cases.
